# Documentation Sync Rules

소스코드와 기능 문서를 병렬로 유지하기 위한 규칙입니다.

---

## 핵심 원칙

**문서는 코드의 기능 지도입니다.**

- 문서는 코드를 다시 설명하지 않습니다.
- 문서는 기능의 역할, 경계, 위치를 빠르게 이해하게 해야 합니다.
- 문서는 짧게 유지하세요.
- README와 ADR은 한국어로 작성하세요.
- 작업과 직접 관련 없는 README, ADR, 규칙 파일은 읽지 마세요.
- 필요한 경우에만 가장 가까운 관련 문서를 확인하세요.

---

## 기본 구조

소스코드와 문서는 섞지 않습니다.

- `src/`에는 실제 소스코드를 둡니다.
- `docs/`에는 기능 설명 문서를 둡니다.
- `docs/`는 `src/`의 기능 경로를 병렬로 따라갑니다.
- 소스 폴더에 직접 `.md` 파일을 섞지 마세요.

예:

- `src/prd/research/`
- `docs/prd/research/README.md`

---

## 목표/디자인 문서 동기화

`goal/`은 현재 제품 목표와 디자인 원본의 위치입니다.

- `goal/DAILY_REPORT_BUILD_GOAL.md`는 요구사항과 완료 기준의 기준 문서입니다.
- `goal/Daily Report.html`은 Claude Design 원본입니다. 사용자가 새 export를 제공하지 않는 한 직접 수정하지 마세요.
- `goal/README.md`는 작업 전 읽을 문서, 문서 우선순위, 디자인/목표 문서 역할을 설명하는 인덱스입니다.
- `goal/CLAUDE_CODE_GOAL_PROMPT.md`는 Claude Code `/goal` 실행 프롬프트입니다.

목표 문서 갱신 기준:

- 사용자 요구사항 자체가 바뀌면 `DAILY_REPORT_BUILD_GOAL.md`를 갱신하세요.
- 구현상 보류하거나 단순화한 항목은 목표 문서를 고쳐 숨기지 말고, 기능 README나 알려진 제한에 기록하세요.
- 작업 전 읽기 순서, 충돌 판단, 검증 기준, 개발 후 문서 갱신 기준이 바뀌면 `goal/README.md`와 `/goal` 프롬프트를 함께 갱신하세요.

디자인 원본과 구현이 달라졌을 때:

- Claude Design HTML은 원본으로 유지하세요.
- 의미 있는 UI/UX 차이는 관련 기능 README 또는 개발 보고에 이유와 함께 기록하세요.
- 디자인 방향 자체를 바꾸는 장기 결정이면 ADR 필요 여부를 검토하세요.

## README 생성 기준

README는 기능 경계가 있는 폴더에만 만드세요.

만드는 경우:

- 독립적인 기능 책임을 가진 폴더
- 하위 기능 폴더를 포함한 폴더
- 여러 파일이 협력해 하나의 기능을 이루는 폴더
- 다른 기능에서 참조할 가능성이 있는 폴더

만들지 않는 경우:

- 단순 정리용 폴더
- 임시 폴더
- 기능 책임이 없는 보조 폴더
- 파일명만으로 역할이 충분히 명확한 경우

---

## README 내용

README에는 다음만 짧게 기록하세요.

- 역할
- 책임 범위
- 포함된 하위 기능
- 주요 파일
- 기능 의미 변경 로그

쓰지 마세요.

- 코드의 줄 단위 설명
- 함수 내부 구현 설명
- 사소한 변수명 설명
- 긴 개발 일지
- git history를 반복하는 변경 기록

템플릿은 `docs/templates/feature-readme.md`를 따르세요.

---

## 상위 README와 하위 README

상위 README는 하위 기능을 자세히 설명하지 않습니다.

상위 README는 다음만 기록합니다.

- 상위 기능의 역할
- 직접 가진 주요 파일
- 하위 기능 목록과 한 줄 설명

하위 README는 해당 기능의 역할과 책임만 기록합니다.

---

## 문서 갱신 기준

모든 변경마다 문서를 갱신하지 마세요.

문서를 갱신해야 하는 경우:

- 기능의 역할 변경
- 기능 흐름 변경
- 입출력 변경
- 의존성 변경
- 하위 기능 추가 또는 제거
- 폴더 구조 변경
- 주요 파일 추가 또는 제거

문서를 갱신하지 않아도 되는 경우:

- 포맷팅 변경
- 변수명 변경
- import 정리
- 내부 구현 정리
- 동작이 바뀌지 않은 리팩터링
- 테스트 코드의 사소한 정리

변경 로그는 “코드 변경 로그”가 아니라 “기능 의미 변경 로그”입니다.  
한 줄로만 작성하세요.

예:

- 2026-05-04: PRD 리서치 결과를 리포트 생성 흐름에 연결

---

## 문서 갱신 위치

소스 파일을 변경했다면 가장 가까운 기능 README를 갱신하세요.

예:

- `src/prd/research/collect.ts` 변경
- `docs/prd/research/README.md` 갱신

하위 기능 변경이 상위 기능 흐름에도 영향을 주면 상위 README도 갱신하세요.

틀린 문서는 없는 문서보다 더 위험합니다.
