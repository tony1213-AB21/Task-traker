# ADR Rules

큰 아키텍처 결정을 기록하기 위한 규칙입니다.

---

## 핵심 원칙

**되돌리기 어렵거나 비용이 큰 설계 결정만 ADR로 기록하세요.**

README는 현재 기능 구조를 설명합니다.  
ADR은 왜 그런 구조를 선택했는지 설명합니다.

---

## ADR이 필요한 경우

다음 조건을 모두 만족하면 ADR을 작성하세요.

- 여러 기능 또는 장기 유지보수에 영향을 준다.
- 나중에 되돌리기 어렵거나 비용이 크다.
- 선택 이유와 트레이드오프를 남길 가치가 있다.

예:

- 핵심 라이브러리 또는 프레임워크 선택/교체
- 데이터 흐름이나 의존성 방향의 큰 변경
- 외부 시스템 연동 방식 결정
- 인증/권한 모델 변경
- 저장소, DB, 캐시, 큐 같은 인프라 선택
- 모노레포, 패키지 경계, 배포 단위 변경
- 기능 간 공통 모듈 분리 방식 결정
- AI 에이전트가 따르는 문서 우선순위나 개발 준비 문서 체계 변경

---

## ADR이 필요 없는 경우

ADR로 남기지 마세요.

- 사소한 버그 수정
- 변수명 변경
- 파일명 정리
- 포맷팅 변경
- 단일 기능 내부의 작은 구현 변경
- 테스트 코드 추가
- 작은 README 변경
- 쉽게 되돌릴 수 있는 결정

---

## ADR 위치와 형식

ADR은 `docs/adr/`에 작성합니다.

파일명:

- `0001-short-decision-name.md`
- `0002-short-decision-name.md`

템플릿은 `docs/templates/adr.md`를 따르세요.

ADR은 한국어로, 짧게 유지하세요.
