# Architecture Rules

기능 경계, 캡슐화, 모듈화, 폴더 구조에 대한 수동 참조 규칙입니다.

새 폴더를 만들거나 폴더 구조를 바꿀 때 확인하세요.  
단순 버그 수정, 변수명 변경, 작은 내부 구현 변경에는 읽지 않아도 됩니다.

---

## 현재 저장소 적용

- 현재 저장소는 Daily Report 앱 개발 전 준비 문서가 중심입니다.
- `goal/`은 현재 제품 목표와 Claude Design 원본의 위치로 유지합니다.
- `goal/Daily Report.html`은 디자인 기준 원본이므로 앱 소스나 `public/`으로 옮겨 구현물처럼 다루지 마세요.
- 앱 소스가 이미 있다면 기존 구조를 먼저 이해하고 유지하세요.
- 앱 소스가 없다면 `goal/DAILY_REPORT_BUILD_GOAL.md`의 추천 구조를 참고하되, 프레임워크 관례 안에서도 기능 의미가 드러나게 만드세요.
- 문서 전용 구조는 `docs/`, `.claude/`, `goal/`의 역할을 섞지 않습니다.

---

## 핵심 원칙

**기능은 나누고, 감추고, 구조로 드러내세요.**

- 프레임워크 관례를 먼저 존중하세요.
- 그 위에서 기술 종류보다 기능 의미를 우선하세요.
- 한 기능의 변경이 다른 기능을 깨뜨리지 않게 하세요.
- 폴더 트리만 봐도 전체 아키텍처가 읽히게 하세요.

---

## 캡슐화

- 기능 내부 구현은 가능한 한 그 기능 안에 가두세요.
- 다른 기능이 내부 파일이나 내부 상태에 직접 의존하지 않게 하세요.
- 기능 바깥으로 노출되는 진입점은 최소화하세요.
- 결합을 줄일 수 있다면 작은 중복은 허용하세요.
- 공통화 때문에 기능 경계가 흐려진다면 공통화를 미루세요.

---

## 모듈화

- 큰 기능은 의미 있는 작은 기능 단위로 나누세요.
- 각 모듈은 하나의 명확한 책임을 가져야 합니다.
- 하나의 모듈이 너무 많은 일을 하면 하위 기능으로 나누세요.
- 기능 간 연결은 명확한 인터페이스를 통해 이루어져야 합니다.
- 내부 구현을 재사용하려고 다른 기능의 내부 폴더를 직접 참조하지 마세요.

---

## 폴더 트리

폴더 구조는 아키텍처의 목차처럼 읽혀야 합니다.

- 상위 폴더는 제품의 큰 기능을 나타내세요.
- 하위 폴더는 그 기능을 이루는 작은 기능을 나타내세요.
- 최하단 파일명은 그 기능 안에서의 구체적 책임을 드러내세요.
- 폴더 이름은 구현 방식보다 기능 의미를 먼저 드러내야 합니다.

선호하는 방향:

- `prd/`
- `prd/research/`
- `prd/report/`
- `auth/login/`
- `auth/session/`

주의할 방향:

- `components/`
- `hooks/`
- `utils/`
- `services/`
- `helpers/`
- `lib/`

위 폴더들이 항상 나쁜 것은 아닙니다.  
Next.js, React, Nuxt 등 프레임워크나 생태계 관례가 요구한다면 그것을 우선 따르세요.

예:

- Next.js의 `app/`, `pages/`, `public/`
- React 프로젝트의 `components/`
- 테스트 도구가 요구하는 `__tests__/`

다만 관례적 폴더 안에서도 기능 의미가 드러나야 합니다.

예:

- `components/prd/`
- `components/auth/login/`
- `hooks/auth/session/`
- `services/payment/refund/`

---

## 공통화 기준

공통화해도 되는 경우:

- 공통 코드가 독립적인 의미를 가진다.
- 공통화해도 기능 경계가 흐려지지 않는다.
- 변경 시 영향 범위를 명확히 예측할 수 있다.
- 이름이 `utils`, `helper`, `manager`처럼 모호하지 않다.

공통화를 미뤄야 하는 경우:

- 단순히 코드 모양이 비슷할 뿐이다.
- 각 기능의 변화 방향이 다를 가능성이 높다.
- 공통화하면 한 기능 변경이 다른 기능에 영향을 준다.
- 공통 모듈 이름이 모호해진다.

작은 중복은 괜찮습니다.  
흐려진 기능 경계는 나쁩니다.
