# AGENTS.md

이 파일은 이 프로젝트의 AI 코딩 원칙에 대한 단일 출처입니다.

Claude Code와 Codex는 이 파일을 기준으로 같은 개발 원칙을 따릅니다.  
Claude Code는 `CLAUDE.md`에서 `@AGENTS.md`로 이 파일을 가져옵니다.  
Codex는 `AGENTS.md`를 저장소 지침으로 직접 읽습니다.

원칙을 추가하거나 수정할 때는 이 파일에서만 변경하세요.  
`CLAUDE.md`에는 공통 코딩 원칙을 중복해서 적지 마세요.

**트레이드오프:**  
이 지침은 속도보다 신중함을 우선합니다.  
단순한 작업에는 상황에 맞게 판단하세요.

---

## 보조 규칙 파일

일부 세부 규칙은 별도 파일에 있습니다.

- `.claude/docs/architecture.md`: 폴더 구조, 기능 경계, 캡슐화, 모듈화 판단
- `.claude/rules/docs-sync.md`: 기능 README 생성/갱신 판단
- `.claude/rules/adr.md`: ADR 작성 여부 판단
- `goal/README.md`: 현재 제품 목표, 개발 목표, 디자인 원본, 문서 우선순위, `/goal` 프롬프트 위치

규칙 파일은 `.claude/` 아래에 있지만, Codex도 필요할 때 직접 읽고 따르세요.  
단, `.claude/rules/`의 자동 path-scoped 로드는 Claude Code 전용 메커니즘입니다.

현재 제품 개발 또는 `/goal` 실행과 관련된 작업에서는 `goal/README.md`를 작업과 직접 관련된 문서로 봅니다.

작업과 직접 관련 없는 README, ADR, 규칙 파일은 읽지 마세요.

## 1. 코딩하기 전에 먼저 생각하기

**추측하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 드러내세요.**

다음 중 하나라도 해당하면 구현 전에 멈추고 확인하세요.

- 요청이 두 가지 이상으로 해석될 수 있다.
- 파일 5개 이상에 영향을 준다.
- 기존 동작을 바꾼다.
- 데이터 모델, API 시그니처, 공개 인터페이스를 바꾼다.
- 되돌리기 어려운 구조 변경을 포함한다.

그 외에는 가정을 한 줄로 명시하고 바로 진행하세요.

## 2. 단순함을 최우선으로

**문제를 해결하는 최소한의 코드만 작성하세요.**

- 요청받은 것 이상의 기능을 추가하지 마세요.
- 한 번만 쓰는 코드에 추상화를 만들지 마세요.
- 요청받지 않은 유연성이나 설정 가능성을 추가하지 마세요.
- 불필요한 옵션, 계층, 방어 코드를 만들지 마세요.
- 단순한 해결책이 있다면 복잡한 설계를 선택하지 마세요.

스스로에게 물어보세요.

> 시니어 엔지니어가 이 코드를 과하게 복잡하다고 말할까?

그렇다면 단순화하세요.

## 3. 외과수술처럼 정확하게 변경하기

**반드시 필요한 부분만 건드리세요.**

기존 코드를 수정할 때:

- 주변 코드, 주석, 포맷팅을 개선하려고 하지 마세요.
- 고장 나지 않은 것을 리팩터링하지 마세요.
- 기존 스타일을 따르세요.
- 관련 없는 죽은 코드는 언급만 하고 삭제하지 마세요.

정리해야 하는 것:

- 이번 작업에서 본인이 마지막 사용처를 제거한 import, 변수, 함수는 같이 제거하세요.
- 기존에 이미 존재하던 죽은 코드는 요청받지 않는 한 제거하지 마세요.

**검증 기준:**  
변경된 모든 줄은 사용자의 요청과 직접 연결되어 있어야 합니다.

## 4. 목표 중심 실행

**성공 기준을 정의하고, 검증될 때까지 반복하세요.**

작업을 검증 가능한 목표로 바꾸세요.

- “검증 로직 추가” → 잘못된 입력 테스트 작성 후 통과
- “버그 수정” → 버그 재현 테스트 작성 후 통과
- “리팩터링” → 리팩터링 전후 테스트 통과 확인

여러 단계가 필요한 작업이라면 짧은 계획을 제시하세요.

- 1단계: 무엇을 할지 → 검증 방법
- 2단계: 무엇을 할지 → 검증 방법
- 3단계: 무엇을 할지 → 검증 방법

## 5. 기능 경계를 우선하기

**기능은 나누고, 감추고, 구조로 드러내세요.**

- 한 기능의 변경이 다른 기능을 깨뜨리지 않게 하세요.
- 큰 기능은 의미 있는 작은 기능 단위로 나누세요.
- 폴더 구조는 아키텍처의 목차처럼 읽혀야 합니다.
- 프레임워크 관례를 우선 따르되, 그 안에서 기능 의미가 드러나게 하세요.
- 공통화 때문에 기능 경계가 흐려진다면 작은 중복을 허용하세요.

새 폴더를 만들거나 기능 경계 판단이 필요하면 `.claude/docs/architecture.md`를 확인하세요.

---

## 6. 기능 문서를 소스와 병렬로 유지하기

**문서는 코드의 기능 지도입니다. 짧고, 병렬적이고, 최신이어야 합니다.**

- `src/`의 기능 구조는 `docs/`에 같은 경로로 문서화하세요.
- 소스 폴더에 직접 `.md` 파일을 섞지 마세요.
- 모든 변경마다 문서를 갱신하지 마세요.
- 기능의 역할, 흐름, 입출력, 의존성, 구조가 바뀐 경우에만 문서를 갱신하세요.
- README와 ADR은 한국어로 작성하세요.

README 생성 또는 갱신 여부가 애매하면 `.claude/rules/docs-sync.md`를 확인하세요.

---

## 7. 큰 설계 결정은 ADR로 남기기

**되돌리기 어렵거나 비용이 큰 아키텍처 결정만 짧게 기록하세요.**

- 큰 구조 변경은 폴더 README가 아니라 ADR로 기록하세요.
- ADR은 `docs/adr/`에 작성하세요.
- 사소한 구현 변경은 ADR로 남기지 마세요.
- 핵심 의존성, 프레임워크, 외부 연동 방식을 선택하거나 바꿀 때는 ADR 필요 여부를 검토하세요.

ADR 작성 여부가 애매하면 `.claude/rules/adr.md`를 확인하세요.

완료 전에 내부적으로만 확인하세요.

- 요청받은 범위만 변경했는가?
- 불필요한 추상화나 기능을 추가하지 않았는가?
- 기능 경계와 폴더 구조가 명확한가?
- 필요한 테스트 또는 수동 검증을 했는가?
- 기능 의미가 바뀌었다면 필요한 문서나 ADR을 갱신했는가?
